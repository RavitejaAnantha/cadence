# Causal attribution: why an offline metric is not a causal claim

Cadence's harness reports NDCG@5 against a synthetic ground-truth relevance, and on that
metric the personalized variant beats the popularity baseline. That is an offline,
measurement result. It is not evidence that deploying the personalized recommender would
cause more engagement online. This file says why, and what we would do to earn a causal claim.

## What the harness actually measures

- **NDCG@5 vs true relevance**: ranking quality against a known utility function. This is
  only possible because the data is synthetic and we generated the utility. In production
  you never have this label.
- **Recall@5 of logged clicks**: the metric you can compute from production logs. The
  harness shows it disagrees with NDCG and tends to flatter the popularity baseline.

## Three reasons offline is not causal

1. **Exposure confounding (the big one).** The logs were generated by a logging policy that
   shows popular tracks more often. Clicks therefore reflect what was shown, not what is
   most relevant. Scoring a new policy on these logs rewards agreeing with the old policy.
   This is off-policy evaluation, and naive replay is biased.
2. **Position bias.** Click probability in the generator is discounted by rank position, so
   a relevant title shown at position 9 is underrepresented in clicks.
3. **Feedback loops and non-stationarity.** A deployed recommender changes what users see,
   which changes future logs. The offline estimate assumes a fixed world. It is not fixed.

## The synthetic ground truth is itself a stand-in

We set true relevance to `0.6 * genre_affinity + 0.4 * energy_fit`. That is an assumption,
not a measurement. Reporting NDCG against it proves the code optimizes the assumed utility,
not that the utility is correct. The real-world version of this risk is trusting a proxy
label (a click, a 30-second play) as if it were the true objective.

## How we would earn a causal claim

- **Randomized A/B test.** The gold standard. Randomize users to baseline vs variant,
  measure primary and guardrail metrics, report effect sizes with CIs, pre-register the
  analysis. The same discipline of reporting bootstrap CIs on everything applies here.
- **Off-policy estimation** when an A/B is too costly. Use the logging propensities the
  generator records (`exposure_prob`) with inverse-propensity scoring (IPS), self-normalized
  IPS, or doubly-robust estimators, and report effective sample size and variance honestly.
  The propensities are recorded precisely so this is possible.
- **Interleaving** for relative ranking quality. Lower variance than a full A/B.
- **Holdout or switchback** for temporal and marketplace effects.

## Guardrails are not optional

Even a real online win can hide harm: collapsed diversity, popular-item runaway, latency
regressions. The harness reports coverage, intra-list diversity, and latency next to the
headline metric. A change that wins NDCG but craters coverage does not ship without a
conversation. The same idea drives a promotion gate that checks multiple-testing-corrected,
out-of-sample evidence and never promotes on one flattering number.

## One-line summary

Offline NDCG tells you the ranker optimizes the objective you gave it. Whether that
objective, deployed, causes the outcome you care about is a separate, experimental
question. Cadence is built to make that distinction impossible to paper over.
